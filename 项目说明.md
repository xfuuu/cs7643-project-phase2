# AI Crime Mystery Story Generation System

一个面向课程项目提交的、可运行的“结构化犯罪推理故事生成系统”。

这份 README 不是面向普通用户的宣传文档，而是面向开发者 / 学生本人，用来解释当前仓库**到底实现了什么、如何运行、数据如何流动、哪些地方是简化实现、哪些地方仍有局限**。

本文严格基于当前仓库代码编写，不描述仓库中不存在的模块、接口或行为。

---

## 1. 项目概述

### 1.1 这个项目要解决什么问题

这个项目尝试解决的问题不是“让大模型直接写一篇侦探小说”，而是：

- 先生成一个**隐藏的案件真相**
- 再把案件真相转换为**结构化事实**
- 再生成**结构化调查计划**
- 再对调查计划做**确定性校验**
- 必要时进行**局部修复**
- 最后把结构化计划实现为**可读的故事文本**

它关注的不只是生成结果“像不像小说”，还关注：

- 真相是否结构清晰
- 证据链是否显式存在
- 调查过程是否可验证
- 输出是否能被后续模块继续处理

### 1.2 这个系统的整体思路

系统的主线由三层核心中间表示组成：

1. `CaseBible`
   - 隐藏真相层
   - 保存案件的 ground truth，包括 investigator、victim、culprit、suspects、motive、method、true_timeline、evidence_items、red_herrings、culprit_evidence_chain

2. `FactTriple`
   - 事实图层
   - 把 `CaseBible` 编译成机器可检查的事实三元组

3. `PlotPlan`
   - 调查过程层
   - 用一组结构化 `PlotStep` 表示“侦探如何一步步逼近真相”

最后，`StoryRealizer` 再把 `PlotPlan` 转换成自然语言故事。

### 1.3 为什么它不是单纯文本生成

这个系统不是纯 prompt -> prose 的一跳式生成，原因在于：

- 上游先生成结构化 `CaseBible`
- 中间再生成结构化 `FactTriple`
- 调查过程先表达为结构化 `PlotPlan`
- `validator.py` 会检查计划是否满足项目要求
- `repair_operator.py` 可以在失败时局部修补
- 文本只是最后一步 realization

所以它的核心思想是：

**结构化生成 + 确定性验证 + 局部修复 + 最终叙事实现**

## 2. 项目整体流程

### 2.1 从入口到输出的完整流程

当前系统的真实执行顺序如下：

```text
python main.py
  ->
main.py 解析命令行参数
  ->
CrimeMysteryPipeline(...)
  ->
pipeline.run()
  ->
CaseBibleGenerator.generate()
  ->
CaseBible
  ->
FactGraphBuilder.build(case_bible)
  ->
fact_graph
  ->
PlotPlanner.build_plan(case_bible, fact_graph)
  ->
initial_plot_plan
  ->
PlotPlanValidator.validate(case_bible, initial_plot_plan)
  ->
if invalid:
    PlotPlanRepairOperator.repair(case_bible, initial_plot_plan, report)
    ->
    PlotPlanValidator.validate(case_bible, repaired_plan)
  ->
StoryRealizer.realize(case_bible, final_plot_plan)
  ->
保存 outputs/*.json 和 outputs/story.txt
```

### 2.2 用“输入 -> 中间表示 -> 校验 -> 修复 -> 输出”的方式理解

可以把整个 pipeline 理解为：

```text
输入
  - generators/setting.txt
  - Gemini / Mock LLM backend

隐藏真相层
  - CaseBible

结构化事实层
  - fact_graph: list[FactTriple]

调查计划层
  - PlotPlan

校验层
  - ValidationReport

修复层
  - repaired PlotPlan（如果初始 plan 不通过）

最终输出层
  - story.txt
  - 各种 JSON 结构化文件
```

### 2.3 `main.py` 和 `pipeline.py` 如何驱动全流程

[main.py](/Users/yuezhao/Documents/New%20project/main.py) 很轻，只做两件事：

- 解析 `--output-dir` 和 `--seed`
- 创建 `CrimeMysteryPipeline` 并调用 `run()`

[pipeline.py](/Users/yuezhao/Documents/New%20project/pipeline.py) 是真正的编排中心，它负责：

- 初始化 Gemini / Mock backend
- 初始化生成、构建、规划、校验、修复、实现模块
- 组织调用顺序
- 把中间结果保存到 `outputs/`

### 2.4 当前版本的真实特点

当前版本不是早期的“调试模式”了，而是完整跑通：

- `CaseBible` 生成
- `fact_graph` 构建
- `plot_plan` 生成
- `validator` 校验
- `repair` 条件触发
- `story` 实现
- 保存全部输出文件

这一点和早期调试阶段不同，README 以当前代码为准。

### [models.py](/Users/yuezhao/Documents/New%20project/models.py)

项目的 schema 中心。  
所有主要模块之间传递的数据对象几乎都定义在这里，包括：

- `TimelineEvent`

如果要理解全项目的数据流，这个文件是最重要的起点之一。

### [llm_interface.py](/Users/yuezhao/Documents/New%20project/llm_interface.py)

统一 LLM 接口层。

这个文件定义了：

- `LLMBackend`
  - 抽象接口
- `MockLLMBackend`
  - 本地可运行的 mock backend
- `GeminiLLMBackend`
  - 真实调用 Gemini API 的 backend

这样上层模块只依赖 `generate(prompt) -> LLMResponse`，不直接依赖某个具体供应商。

### [generators/case_bible_generator.py](/Users/yuezhao/Documents/New%20project/generators/case_bible_generator.py)

负责生成隐藏真相 `CaseBible`。

当前实现已经不是手工写死案件，而是：

1. 读取 [setting.txt](/Users/yuezhao/Documents/New%20project/generators/setting.txt)
2. 构造严格 JSON prompt
3. 调用 LLM 生成 case blueprint
4. 本地解析 JSON
5. 转成 dataclass `CaseBible`

它是整个系统最上游的生成器。

### [generators/setting.txt](/Users/yuezhao/Documents/New%20project/generators/setting.txt)

外部设定文件。

它不是简单的一句 setting 描述，而是一份案件生成约束说明，包括：

- 年代与技术限制
- 封闭庄园 / closed-circle 设定
- 侦探必须在案发前已被邀请在场
- fair-play 推理风格
- 至少 4 suspects、8 evidence、1 red herring、长调查弧线等要求

### [builders/fact_graph_builder.py](/Users/yuezhao/Documents/New%20project/builders/fact_graph_builder.py)

负责把 `CaseBible` 编译成 `FactTriple` 列表。

它做的不是自由生成，而是确定性转换：

- 人物事实
- 动机 / 手法事实
- timeline facts
- evidence facts
- red herring facts

它还会从 `true_timeline` 中推断一些关键时间，如：

- 受害者死亡时间
- 真凶实施方法的时间
- 嫌疑人的活动时间窗口

### [planners/plot_planner.py](/Users/yuezhao/Documents/New%20project/planners/plot_planner.py)

负责生成 `PlotPlan`。

当前实现不是单一模式，而是：

- **LLM 优先**
  - 先要求 Gemini 输出结构化 JSON plot plan
- **规则 fallback**
  - 如果 LLM 失败、JSON 不合法或解析异常，则回退到 rule-based planner

为了避免 LLM 在时间标注上漂移，这个文件现在还包含**本地时间后处理**，会把步骤时间重新锚定到案件发现时间附近，修正夜间案件被写成 `AM` 的问题。

### [validators/validator.py](/Users/yuezhao/Documents/New%20project/validators/validator.py)

负责做确定性结构校验。

它不是评价文学性，而是检查：

- suspects 数量
- evidence 数量
- plot step 数量
- alibi / red herring / interference 是否存在
- confrontation 是否存在且引用关键证据
- evidence id 是否合法
- timeline 是否单调递增

### [repair/repair_operator.py](/Users/yuezhao/Documents/New%20project/repair/repair_operator.py)

负责在 validation 失败时动态补丁。

它不是完全重写 plot，而是根据：

- `CaseBible`
- 现有 `PlotPlan`
- `ValidationReport`

动态补缺，例如：

- 缺少 alibi check
- 缺少 red herring
- 缺少 interference
- culprit evidence chain 未覆盖
- confrontation 缺失
- 步数不足

### [realization/story_realizer.py](/Users/yuezhao/Documents/New%20project/realization/story_realizer.py)

负责把结构化计划实现成故事文本。

它支持两种路径：

- `MockLLMBackend`
  - 简单拼接结构化信息
- `GeminiLLMBackend`
  - 通过 prompt 把 `CaseBible + PlotPlan` 转成更自然的短篇推理叙事

### [pipeline.py](/Users/yuezhao/Documents/New%20project/pipeline.py)

总编排中心。

当前真实行为是：

- `CaseBibleGenerator` 使用 `GeminiLLMBackend`
- `PlotPlanner` 使用 `GeminiLLMBackend`
- `StoryRealizer` 使用 `GeminiLLMBackend`
- 同时仍实例化 `MockLLMBackend`，方便未来切换或调试

### [main.py](/Users/yuezhao/Documents/New%20project/main.py)

命令行入口。

只负责：

- 解析参数
- 创建 pipeline
- 调用 `run()`
- 打印生成结果路径与基础信息

## 4. 核心数据结构 / Schema

### 4.1 `Character`

定义在 [models.py](/Users/yuezhao/Documents/New%20project/models.py)。

字段：

- `name`
  - 人物名
- `role`
  - `victim` / `suspect` / `culprit`
- `description`
  - 人物概况
- `relationship_to_victim`
  - 与死者关系
- `means`
  - 作案手段或潜在手段
- `motive`
  - 动机
- `opportunity`
  - 作案机会
- `alibi`
  - 不在场说法

使用位置：

- `CaseBibleGenerator` 生成并构造
- `FactGraphBuilder` 提取人物事实
- `PlotPlanner` 组织嫌疑人调查顺序
- `StoryRealizer` 将这些人物信息转成叙事压力

### 4.2 `EvidenceItem`

字段：

- `evidence_id`
- `name`
- `description`
- `location_found`
- `implicated_person`
- `reliability`
- `planted`

作用：

- 表示结构化证据
- 在 `fact_graph` 中转成三元组
- 在 `plot_plan` 中用 `evidence_ids` 引用
- 在 confrontation 中形成关键证据链

### 4.3 `TimelineEvent`

字段：

- `event_id`
- `time_marker`
- `summary`
- `participants`
- `location`
- `public`

作用：

- 表示隐藏真相层的真实时间线
- `FactGraphBuilder` 从这里推时间
- `PlotPlanner` 从这里提 discovery / death / concealment / tension 等事件

### 4.4 `RedHerring`

字段：

- `herring_id`
- `suspect_name`
- `misleading_evidence_ids`
- `explanation`

作用：

- 显式表示误导弧线
- `PlotPlanner` 会尝试把它纳入调查计划
- `validator` 会检查是否至少有一个 `red_herring`
- `repair` 可以在缺失时补一条 red herring step

### 4.5 `CaseBible`

这是项目最核心的数据对象。

字段：

- `setting`
- `investigator`
- `victim`
- `culprit`
- `suspects`
- `motive`
- `method`
- `true_timeline`
- `evidence_items`
- `red_herrings`
- `culprit_evidence_chain`

它的含义是：

- 保存案件的隐藏真相
- 与最终调查过程和故事文本严格区分

数据流位置：

```text
setting.txt + LLM
  ->
CaseBibleGenerator
  ->
CaseBible
  ->
FactGraphBuilder / PlotPlanner / StoryRealizer / Validator / Repair
```

### 4.6 `FactTriple`

字段：

- `subject`
- `relation`
- `object`
- `time`
- `source`

注意：

- 当前版本已经删除了早期的 `confidence`
- 现在 `FactTriple` 更简洁，强调结构性而不是分数打分

作用：

- 提供机器可处理的事实图层
- 当前主要被 `PlotPlanner` 用作证据存在性辅助，而不是深层推理引擎

### 4.7 `PlotStep`

字段：

- `step_id`
- `phase`
- `kind`
- `title`
- `summary`
- `location`
- `participants`
- `evidence_ids`
- `reveals`
- `timeline_ref`

作用：

- 表示调查过程中的一个结构化剧情节点
- 是 `PlotPlan` 的基本单元
- 也是 `validator` 校验的直接对象

### 4.8 `PlotPlan`

字段：

- `investigator`
- `steps`

作用：

- 表示侦探的结构化调查计划
- 是 `StoryRealizer` 的直接输入
- 是 `validator` / `repair` 的直接工作对象

### 4.9 `ValidationIssue` 与 `ValidationReport`

`ValidationIssue`：

- `code`
- `message`
- `step_id`

`ValidationReport`：

- `is_valid`
- `issues`
- `metrics`

作用：

- 报告 `PlotPlan` 是否满足课程要求
- 给 `repair_operator.py` 提供修复依据

## 5. 各模块详细说明

### 5.1 `models.py`

主要职责：

- 统一定义整个项目的数据结构

输入：

- 无运行时输入，它提供 schema

输出：

- 所有 dataclass 类型定义

内部逻辑：

- 无复杂逻辑，主要是 schema
- 另有一个 `to_data()` 用于把 dataclass 递归转为普通 Python 数据

依赖：

- 标准库 `dataclasses`, `typing`

被谁调用：

- 几乎所有模块

系统位置：

- 项目的数据基础层

### 5.2 `llm_interface.py`

主要职责：

- 为项目提供统一 LLM 调用接口

关键类：

- `LLMBackend`
- `LLMResponse`
- `MockLLMBackend`
- `GeminiLLMBackend`

输入：

- 一个 prompt 字符串

输出：

- `LLMResponse(text=...)`

内部逻辑：

- `MockLLMBackend` 根据 prompt 关键词返回预设短文本
- `GeminiLLMBackend` 使用 `urllib.request` 调 Gemini API

依赖：

- 标准库 `json`, `urllib`, `os`

被谁调用：

- `CaseBibleGenerator`
- `PlotPlanner`
- `StoryRealizer`
- `pipeline.py`

系统位置：

- 模型调用基础设施层

实现备注：

- 当前 `GeminiLLMBackend` 默认构造函数中仍有硬编码 API key，这对课程项目本地运行方便，但并不是生产级写法
- 更合理的做法是优先从环境变量读取

### 5.3 `generators/case_bible_generator.py`

主要职责：

- 生成隐藏真相 `CaseBible`

输入：

- `setting.txt`
- 一个 `LLMBackend`

输出：

- `CaseBible`

内部核心逻辑：

1. 读取 `setting.txt`
2. 构造严格 JSON prompt
3. 调用 `llm.generate(prompt)`
4. 从返回文本中抽取 JSON 对象
5. 检查顶层 shape 是否齐全
6. 逐字段构造成：
   - `Character`
   - `TimelineEvent`
   - `EvidenceItem`
   - `RedHerring`
7. 最终拼成 `CaseBible`

依赖：

- `llm_interface.py`
- `models.py`

被谁调用：

- `pipeline.py`

系统位置：

- 全系统的最上游生成模块

实现备注：

- 它不是自由文本生成，而是“LLM 输出 JSON blueprint + 本地解析”
- 这种设计使得上游结果可以继续被后续模块结构化使用

### 5.4 `builders/fact_graph_builder.py`

主要职责：

- 把 `CaseBible` 编译为 `FactTriple` 列表

输入：

- `CaseBible`

输出：

- `list[FactTriple]`

内部核心逻辑：

1. 对 `true_timeline` 做时间排序
2. 推断：
   - victim 时间
   - method 时间
   - suspect 时间窗口
3. 生成：
   - 案件 setting facts
   - culprit / victim facts
   - suspect attribute facts
   - timeline facts
   - evidence facts
   - red herring facts

依赖：

- `models.py`
- 标准库 `re`

被谁调用：

- `pipeline.py`

系统位置：

- `CaseBible` 和 `PlotPlanner` 之间的结构化事实层

实现备注：

- 当前时间推断仍是启发式的，不是 LLM 或逻辑定理证明式推理
- 但已经比最早的硬编码时间版本更合理

### 5.5 `planners/plot_planner.py`

主要职责：

- 生成 `PlotPlan`

输入：

- `CaseBible`
- 可选 `fact_graph`

输出：

- `PlotPlan`

内部核心逻辑：

#### LLM 分支

1. 根据 `CaseBible` 和部分 `fact_graph` 构造 plot prompt
2. 要求模型输出严格 JSON：
   - 顶层为 `{"steps": [...]}`
3. 本地解析每个 `step`
4. 过滤非法 `evidence_ids`
5. 规范化 `step_id`
6. 做时间后处理，修正不合理的 `AM/PM`

#### fallback 分支

如果 LLM 分支失败，则退回到规则版 planner：

- 从 `true_timeline` 中提取 discovery / death / concealment / tension
- 结合 suspects / red herrings / culprit evidence chain
- 生成一套固定骨架但内容动态填充的 `PlotPlan`

依赖：

- `llm_interface.py`
- `models.py`

被谁调用：

- `pipeline.py`

系统位置：

- 从真相层走向调查过程层的核心模块

实现备注：

- 当前这是一个 hybrid planner，而不是纯 LLM 或纯规则系统
- 它的优点是既能提升可读性，又保留 fallback 稳定性

### 5.6 `validators/validator.py`

主要职责：

- 对 `PlotPlan` 做确定性结构校验

输入：

- `CaseBible`
- `PlotPlan`

输出：

- `ValidationReport`

内部核心逻辑：

- 检查 suspects 数量
- 检查 evidence 数量
- 检查 plot step 数量
- 检查 `alibi_check` 数量
- 检查 `red_herring` 是否存在
- 检查 `interference` 是否存在
- 检查 culprit evidence chain 是否被引用
- 检查 confrontation 是否存在且引用关键证据
- 检查 `step_id` 是否连续
- 检查 evidence id 是否真实存在
- 检查时间顺序是否单调

依赖：

- `models.py`

被谁调用：

- `pipeline.py`
- `repair_operator.py` 的调用前提

系统位置：

- `PlotPlan` 之后、`Repair` 之前的质量门控层

实现备注：

- 它擅长结构检查
- 但不擅长检查深层叙事语义或事实微漂移

### 5.7 `repair/repair_operator.py`

主要职责：

- 在 validation 失败时动态打补丁

输入：

- `CaseBible`
- 原始 `PlotPlan`
- `ValidationReport`

输出：

- 修复后的 `PlotPlan`

内部核心逻辑：

- 先清理非法 evidence id
- 根据 issue code 动态补：
  - alibi steps
  - red herring
  - interference
  - evidence chain step
  - culprit support step
  - confrontation
  - minimum steps
- 最后重新编号并在必要时重排时间

依赖：

- `models.py`

被谁调用：

- `pipeline.py`

系统位置：

- `Validator` 之后的局部修复层

实现备注：

- 现在已经不是早期的固定模板补丁

## 8. Validator 和 Repair 的机制详解

### 8.1 Validator 检查了哪些规则

当前 validator 会检查：

- suspects 是否至少 4 个
- evidence 是否至少 8 个
- plot step 是否至少 15 个
- `alibi_check` 是否至少 2 个
- 是否至少有 1 个 `red_herring`
- 是否至少有 1 个 `interference`
- culprit evidence chain 是否在 plan 中被引用
- 是否存在 `confrontation`
- confrontation 是否引用关键证据链前 3 项
- culprit 是否在足够多步骤里被支持
- `step_id` 是否连续
- evidence id 是否真实存在
- `timeline_ref` 是否单调递增

### 8.2 为什么这些规则重要

这些规则直接对应课程项目的核心要求：

- 不是随便写个短案子
- 必须有调查结构
- 必须有红鲱鱼
- 必须有 alibi 验证
- 最终揭露必须有证据链支撑

### 8.3 Repair 如何修复

如果 plan 不通过，repair 会根据 `ValidationReport.issues` 动态补洞，例如：

- 没有足够 alibi check，就从 suspects 中补
- 缺 red herring，就从 `CaseBible.red_herrings` 中补
- 缺 interference，就围绕 culprit 补
- 缺关键 evidence chain，就补证据链 step
- 缺 confrontation，就补 confrontation
- 步数不够，就补额外 evidence/analysis step

### 8.4 修复策略的性质

这些修复本质上是：

- **启发式的**
- **面向结构合规的**
- **不是重新规划整个调查弧线**

所以 repair 的目标是“修到能过”，而不是“修到文学效果最好”。

### 8.5 当前机制的局限

- 它不会重写整套 LLM plan
- 不会优化 prose
- 不会主动修复所有细微事实漂移
- 主要关注“规则缺项”

## 9. LLM / Mock 机制详解

### 9.1 `llm_interface.py` 到底做了什么

它为整个项目提供统一接口：

```python
class LLMBackend:
    def generate(self, prompt: str) -> LLMResponse:
        ...
```

这使得上层模块不需要知道底层到底是：

- mock
- Gemini
- 还是未来别的模型

### 9.2 Mock backend 的角色

- 作为接口占位
- 方便调试某些不需要真实生成质量的流程

### 9.3 为什么默认使用真实 Gemini

当前 `pipeline.py` 中：

- `CaseBibleGenerator` 用 Gemini
- `PlotPlanner` 用 Gemini

- `CaseBible` 已经依赖真实 LLM 输出 JSON blueprint
- `PlotPlan` 现在也有 LLM 动态分支

上层模块不需要大改，因为它们只依赖统一的 `generate(prompt)` 接口。

## 10. 这个项目体现了哪些设计思想

### 10.1 Hidden Truth vs Revealed Investigation

这是当前项目最重要的设计思想之一。

- `CaseBible`
  - 隐藏真相
- `PlotPlan`
  - 调查如何逐步显露真相

也就是说：

- 真相不是直接曝光给读者
- 读者看到的是调查过程，而不是 ground truth 本身

### 10.2 Structure Before Prose

项目先生成：

- `CaseBible`
- `FactTriple`
- `PlotPlan`

最后才生成故事文本。

这保证：

- 系统不是靠 prose “糊出来”
- 结构可以被验证、修复、保存

### 10.3 Deterministic Validation

`validator.py` 体现了“规则检查优先”的思想：

- 不依赖模型评价自己
- 而是明确检查是否满足项目要求

### 10.4 Local Repair Instead of Full Regeneration

`repair_operator.py` 体现的是：

- 不是一失败就整份 plan 重新生成
- 而是局部打补丁

这样更稳定，也更符合工程化思路。

### 10.5 Separation of Concerns

项目的职责划分比较清楚：

- generator 负责隐藏真相
- builder 负责事实图
- planner 负责调查计划
- validator 负责规则校验
- repair 负责修复
- realizer 负责文本输出
- pipeline 负责编排

这也是本项目结构上最值得保留的优点之一。

## 11. 当前实现的简化之处与局限性

这一节很重要，课程提交时应诚实说明。

### 11.1 `FactGraphBuilder` 仍然是启发式

虽然它已经摆脱了最早的硬编码时间，但仍然依赖：

- summary 关键词
- timeline 中的人名与动作描述

它不是严格的语义推理系统。

### 11.2 Validator 主要检查结构，不检查深层语义

它可以发现：

- step 数量不够
- evidence id 不合法
- 缺少 confrontation

但不一定能发现：

- 计划文本和 `CaseBible` 的细微事实漂移
- 某些自然语言细节被 LLM 悄悄改写

### 11.3 Repair 是启发式补丁器，不是再规划器

repair 能补缺项，但不能保证：

- 节奏最自然
- 叙事最优雅
- 所有细微事实都被重新对齐

### 11.4 PlotPlanner 的 LLM 输出仍可能有事实漂移

虽然现在：

- 有 fallback
- 有时间后处理
- 有 validator

但 LLM 仍可能：

- 自己补写一些 `CaseBible` 中没有明确写出的场景细节
- 在个别时间点上做出近似而非精确复述

### 11.5 StoryRealizer 也可能继承上游偏差

如果 `PlotPlan` 有轻微漂移，`story.txt` 往往会把这种漂移叙事化。

### 11.6 Gemini backend 的实现是课程项目级

优点：

- 最小依赖
- 可运行

局限：

- 没有重试机制
- 超时处理较简单
- API key 当前默认值写在代码里，不是最佳实践

### 11.7 Mock 分支质量明显弱于 Gemini 分支

这不是 bug，而是当前设计目标决定的：

- Mock 主要为了可运行
- Gemini 才是当前版本的主要生成质量来源
